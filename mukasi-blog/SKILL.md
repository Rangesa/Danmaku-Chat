# Mukasi Blog

2000年代〜2010年代の個人ブログのような、自然で人間味のある記事を作成するスキル。

## 核心原則

### AI臭を徹底的に排除

以下の表現は**絶対に使用禁止**：

**過剰な丁寧さ**
- 「〜してみてください」「〜してみましょう」
- 「〜かもしれません」の連発
- 「ぜひ〜」の多用

**機械的な構成**
- 必ず「まず」「次に」「最後に」で始まる三段構成
- 箇条書きの乱用（リストが必要な場合を除く）
- 「以下の通りです」「下記の通り」

**不自然な強調**
- 絵文字の使用（✨🎉💡など）
- 中黒「・」による不必要なリスト化
- 過剰な太字や装飾

**浅い内容**
- 検索で出る一般論の羅列
- 具体性のない説明
- 誰にでも言える当たり前の内容

### 人間らしさを出す要素

**顔文字の適切な使用**
- 感情表現：(´・ω・`)、('ω')、(｀･ω･´)
- リアクション：|дﾟ)、( ˘ω˘ )、(ﾉ∀`)
- テンション高い時：ｷﾀ━━━━(ﾟ∀ﾟ)━━━━!!
- 使用頻度：見出しごとに0〜1個程度、自然な場所のみ

**自然な文体**
- 口語的な表現を混ぜる（「〜なんでっすよね」「〜だったりする」）
- 適度な脱線や余談
- 断言と留保のバランス
- 失敗談や試行錯誤の過程を含める

**具体性**
- 実際の数値やスクリーンショット
- 「3日かかった」「最初は○○で失敗した」など具体的なエピソード
- 独自の視点や気づき

## 記事構成パターン

### 自作ツール/MOD紹介記事の基本構成

```
【導入】作った経緯を自然に語る
- 困っていたこと
- 既存の解決策の不満点
- 「だから作った」への自然な流れ

【制作過程】試行錯誤を含める
- 最初の設計
- 実装で躓いた部分
- 解決方法（技術的な詳細を含む）
- 予想外の発見

【完成と使用感】押し付けがましくない紹介
- 実際に使ってみた感想
- 便利だった点
- まだ改善したい点（完璧ではないことを示す）

【共有】軽い宣伝
- 「良かったら使ってみてください」程度
- GitHubリンクなどを自然に配置
```

## 記事作成プロセス

1. **ユーザーから紹介したいものの情報を収集**
   - 何を作ったのか
   - なぜ作ったのか
   - 技術的な特徴
   - 苦労した点

2. **記事の構成を決定**
   - 上記の基本構成を参考に
   - 内容に応じて柔軟に調整

3. **執筆時の注意点**
   - AI臭い表現を一切使わない
   - 顔文字は自然な場所に控えめに
   - 具体的なエピソードを入れる
   - 技術詳細は書くが説教臭くしない
   - 完璧すぎない（改善点も書く）

4. **最終チェック**
   - 禁止表現が含まれていないか確認
   - 不自然な箇条書きになっていないか
   - 人間が書いた文章として違和感がないか

## 具体例

### 例1：Minecraft MOD紹介

```
最近Fabricでユーティリティクライアント作ってた話

Minecraftで遊んでる時に、いちいちF3押してFPS確認するの面倒だなって思ってたんですよね(´・ω・`)

既存のMODだとHUD表示が微妙に見づらかったり、カスタマイズ性に欠けてたりで...

だから自分で作ることにした('ω')

## 実装について

Fabric APIを使って、基本的なHUD描画から始めた。OpenGLのシェーダーで見た目を調整したんだけど、最初は座標計算でかなり躓いた。画面解像度が変わるとレイアウトが崩れるんですよね。

結局ScaledResolutionを使って相対座標で実装し直したら上手くいった。

あと、フルブライト機能も追加してみた。夜間探索が楽になって想像以上に便利|дﾟ)

## 使ってみた感想

自分用に作ったからUIは結構シンプル。でも必要な情報は全部表示されるし、動作も軽い。

まだ設定画面がないのが難点だけど、そのうち実装する予定。

GitHubに上げてあるので、良かったら使ってみてください。
```

### 例2：ツール紹介

```
Claude Chat履歴をエクスポートする拡張機能作った

Claudeのチャット履歴、後で見返したい時に不便だったんですよね。特にコードのやり取りとか。

ブラウザ拡張で何とかできないかなと思って調べてたら、意外と簡単に作れそうだったので挑戦してみた。

## 実装の流れ

Manifest V3で開発。最初はContent Scriptでページ内容を取得しようとしたけど、Claudeが動的にコンテンツ読み込んでるせいで上手くいかない。

MutationObserverを使ってDOM変更を監視する方式に変更したら、ちゃんと全てのメッセージを取得できた(｀･ω･´)

エクスポート形式はMarkdown、JSON、HTMLの3種類に対応させた。

## 今後の予定

今のところChrome専用だけど、Firefoxにも対応させたい。あとバックアップの自動保存機能とかあると便利そう。

需要があるか分からないけど、Chrome Web Storeに公開予定です。
```

## 文体サンプル

**良い例（人間らしい）**
- 「最近○○で困ってたんですよね」
- 「結局△△で解決した」
- 「意外と便利で自分でも驚いた」
- 「まだ改善の余地はある」

**悪い例（AI臭い）**
- 「○○について解説していきます」
- 「以下の手順で実装してみましょう」
- 「ぜひ参考にしてみてください」
- 「いかがでしたでしょうか」
